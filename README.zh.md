## 设计决策

**前端保留一份提醒列表的本地副本**
列表本身很少变动,但应用每秒都要检查是否有提醒到点。每秒向后端问一次既浪费,
网络一抖还会漏检。后端是数据的权威来源,前端这份只是缓存。
*代价:* 如果数据在其他地方被改动(另一台设备、另一个标签页),前端要等到下次刷新才知道。

**写操作用增量接口(POST 一条 / DELETE 一条),而不是整份覆盖**
整份覆盖这个操作,没法反向推出具体删改了什么时间这个细节。所以发送这个删改本身而不是
整个数组，才好明确意图,而且不会随着数据变多而变慢。
*代价:* 接口数量比"一个全量替换"多。

**排序放在后端做**
排序等等规则只由一个地方拥有。两边都排的话,规则一旦不一致,界面显示的顺序就会和
实际存储的悄悄不同。
*代价:* 前端想调整顺序必须走一趟网络。

**`lastAlert` 存在 localStorage,不进数据库**
这个记录某个提醒今天是否已经响过,防止同一分钟内重复提醒。这属于每台设备各自的
运行时状态，因此在电脑上响过,不该算作在手机上也响过。真正值得存到服务端的只有
提醒列表本身。
*代价:* 清空浏览器存储就会重置。

**写操作之后,前端直接采用后端返回的列表,而不是自己计算新状态**
后端负责排序和分配 id,直接采用它的返回值,能保证界面和实际存储完全一致。
*代价:* 每次增删有几百毫秒的可见延迟。其实加上乐观更新可以掩盖延迟,但那意味着要为
失败的请求写回滚逻辑，对这种低频操作不值得。

## 已知的局限

- 定时检查跑在浏览器里,标签页一关就不会再提醒。真正的后台推送需要
  Service Worker + Web Push,或者做成原生应用。
- 没有用户系统。接口无鉴权,所有客户端读写的是同一份列表。
- API 地址和 CORS 来源都硬编码成了 localhost,没有任何可配置项。
- `lastAlert` 会在刷新后重置,所以同一分钟内可能重复提醒一次。
- 建表用的是 `CREATE TABLE IF NOT EXISTS`,没有 migration 机制——改表结构
  只能删掉数据库文件重来。
